# 需求分析

产品需求是走在具体开发前面，只有评审合格后的产品需求才能交付进一步开发。

最重要的是：`产品需求一定是可用的，明确定`。

不要需求都没有确定，就开始着手开发，那样会进入无止境的修改推脱责任中...

把不靠谱（不可用)的需求留在后面做，可能到时候需求就会更改。


## 产品设计5大层面 

首先整个产品设计分为5个大的方面，由低到高，底层根本原因决定上层形态构成。和网站分层也很形象。

1. 战略层：我们为什么要做这个产品？

2. 范围层：我们要建设什么样的内容？

3. 结构层：如何呈现给用户？模式、顺序、优先级？

4. 框架层：哪些功能在哪些页面上完成？这些功能在页面上如何实现？

5. 表现层：不仅是美化，而是产品功能是否得到很好的展示？


## 从前端开发角度整理各个层面需求的评审

#### 1. 战略层：我们为什么要做这个产品？
* 我们要从这个产品中得到什么？产品目标是什么？
* 我们的用户要从这个产品中得到什么？产品需求是什么？

更好地去理解web产品用户群体，产品目标。会对产品设计需求有更好的理解。

理解产品的定位对前端开发技术选型及重要侧重点会有很大影响：

1. 专题制作——上线快，下线快 更加注重时间，开发效率；
2. 邮件页面——兼容性，table布局，稳定性；
3. 平台型产品——在线时间长，团队规范，可维护性，可扩展型；
 

#### 2. 范围层：我们要建设什么样的内容？

需要明确完善的`项目需求文档`，

把用户需求和网站目标转变成网站应该提供给用户什么样的`内容和功能`，避免`范围蠕动`。

知道正在建设什么？也要知道不需要建设什么？

这一点也非常重要，不能人云亦云，盲目跟风。

明确永远不需要建设的，及现在不需要建设的，可为后面迭代版本做准备的。

`项目需求文档`，明确整个项目需要的内容需求。这个也后端数据库也非常相关。

内容需求： 
* 具体需要哪些形式的内容—— 文本、图片、音频、视频

* 具体内容的大小规模——字数范围、像素大小、下载文件字节、音频视频清晰度、 内容更新频率、呈现方式

* 确定需求优先级——用户最希望的内容是什么？实现这些需求的可行性有多大？
