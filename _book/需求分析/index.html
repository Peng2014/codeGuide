需求分析

产品需求是走在具体开发前面，只有评审合格后的产品需求才能交付进一步开发。

最重要的是：产品需求一定是可用的，明确定。

不要需求都没有确定，就开始着手开发，那样会进入无止境的修改推脱责任中...

把不靠谱（不可用)的需求留在后面做，可能到时候需求就会更改。

产品设计5大层面

首先整个产品设计分为5个大的方面，由低到高，底层根本原因决定上层形态构成。和网站分层也很形象。

  1. 战略层：我们为什么要做这个产品？
  2. 范围层：我们要建设什么样的内容？
  3. 结构层：如何呈现给用户？模式、顺序、优先级？
  4. 框架层：哪些功能在哪些页面上完成？这些功能在页面上如何实现？
  5. 表现层：不仅是美化，而是产品功能是否得到很好的展示？

从前端开发角度整理各个层面需求的评审

1. 战略层：我们为什么要做这个产品？

  • 我们要从这个产品中得到什么？产品目标是什么？
  • 我们的用户要从这个产品中得到什么？产品需求是什么？

更好地去理解web产品用户群体，产品目标。会对产品设计需求有更好的理解。

理解产品的定位对前端开发技术选型及重要侧重点会有很大影响：

1. 专题制作——上线快，下线快 更加注重时间，开发效率；
2. 邮件页面——兼容性，table布局，稳定性；
3. 平台型产品——在线时间长，团队规范，可维护性，可扩展型；

2. 范围层：我们要建设什么样的内容？

需要明确完善的项目需求文档，

把用户需求和网站目标转变成网站应该提供给用户什么样的内容和功能，避免范围蠕动。

知道正在建设什么？也要知道不需要建设什么？

这一点也非常重要，不能人云亦云，盲目跟风。

明确永远不需要建设的，及现在不需要建设的，可为后面迭代版本做准备的。

项目需求文档，明确整个项目需要的内容需求。这个也后端数据库也非常相关。

内容需求： 

1.具体需要哪些形式的内容—— 文本、图片、音频、视频
* 明确具体哪些图片是`img`，哪些图片为`background`;
* 明确哪些图片或图标为`ico字体图标`，哪些实际`图片`;
* 明确哪些内容是本质是`文本`，实际展示方式是通过`文字隐藏`，而显示其`背景图`；

2.具体内容的大小规模——字数范围、像素大小、下载文件字节、音频视频清晰度、 
* 明确哪些文本内容字数范围容易变化（防止文字溢出破坏板式）
* 明确不同图片的格式选择
* 不同分辨率下图片大小的选择
* 注意效果图的文案是最终确定的文案吗？

3.内容更新频率、呈现方式
* 哪些数据内容静态处理，哪些数据需要后代动态渲染输出;
* 列表数量具体显示情况

4.确定需求优先级——用户最希望的内容是什么？实现这些需求的可行性有多大？

可能这里前端更具体的是让程序识别出web内容需求的优先级

* 审核需求可行性，具体数据如何传递，如何与后端配合;
* h1~h6的谨慎选择，特别是`h1`,一般用在整个页面`logo`,注意不冲突，页面结构合理
* 注意页面`head`标签内页面`titile`,`keywords`,`description`等设置，
* 注意页面`img`表情的`alt`属性，重要标签的`title`属性

3. 结构层：如何呈现给用户？确定各个将要呈现给用户的选项模式和顺序，优先级？

这个层面主要包括交互设计和信息架构，主要以交互流程图和信息架构图为主。

1.交互设计：明确可能的用户行为

* 逻辑复杂的界面是否有给出明确的交互流程图
* 最终要明确哪些功能该在哪些页面上完成
* 如果用户头像，或图片未传地址，是否需要显示默认图片
* 用户输入，按钮交互，各个步骤及时进行信息提示，(开始，进行中，完成，错误等状态)
* 在不同交互步骤下，如何配合用户执行和完成任务的选项，如：按钮是否可点击
* 当用户犯错时，系统怎么反应，当错误第一次发生时，系统要如何防止继续犯错，最好将系统设计成不可能放错或 难以犯错。
* 最后有重做或撤销功能。删除时也不完全删除，而是放入回收站。
* 页面上哪些地方需要具备a标签跳转功能，是本页跳转还是新开页面跳转？

2.信息架构：组织管理、分类、顺序排列，如何将信息表达给用户的选项，设计组织分类和导航的结构。

创建分类体系，确保用户是否认为每一个步骤都是合理的，当前的步骤是否自然的延续上一个步骤的任务，

产品是有生命的个体，和后端代码设计也有同样的要求，要有很好的健壮性，能够容纳成长和适应变动，等到后期添加内容时不会有太大影响，也能够很好的组合。

结构方法：组织原则，命名原则，都要让用户在整个网站的结构中找到想要走的的路。整个网站要能够很好的保持一致性。要能够很好的使用用户属性的语言规范。


* 注意核实效果图上的文案是否为最终效果；
* 如果有数字金额显示，注意取值范围，单位符号

4. 框架层：哪些功能在哪些页面上完成？这些功能在页面上如何实现？

功能在页面上面如何被实现。

界面设计：让用户一眼就要看到“最重要的东西”

导航设计：提供用户去某个地方的能力

信息设计：创达想法给用户

线框图：界面设计、导航设计、信息设计、三者融合的产物，三者关系相互影响，很多因素都得同时考虑

A.界面设计：弄清用户不需要知道的内容，减少可视性，组织好用户最常采用的行为。让用户完成目标的过程变得容易。

* 仔细考虑好每一个选项的默认值。不同功能控件的选择（按钮，复选框，单选框，文本框，下拉菜单，多选菜单），及时提供错误说明消息。


B.导航设计： 告诉用户在哪里？能去哪里？ 导航设计中必须创达出它的内容和用户当前浏览页面之间的关系。

* 整个导航包括了多个系统：全局导航、局部导航、上下文导航、友好导航、网站地图、索引表等。


C.信息设计：分组和整理散乱的信息，反应用户的思路;

* ，有时信息设计是考视觉上呈现：比如饼状图还是柱状图谁更好呈现数据？


D.线框图：将整个页面布局规划好，必要时参考文档交互设计和内容需求说明

5. 表现层：不仅是美化，而是产品功能是否得到很好的展示？


* 注意效果图上相同模块样式保持统一,如：a标签hover,active等，颜色变化，是否有下划线；在效果图上无法体现，需要整个页面都统一

创造更好的用户体验

五个层面的内容都要有系统的架构认识。

1.了解正在试着去解决的问题属于哪个层？

2.了解解决办法所创造的后果会影响哪个层？

1.了解正在试着去解决的问题属于哪个层？

比如：如果已经知道在主页那个紫色大按钮是个问题？

那么是按钮太大还是紫色不合适？他们哪个需要改变（属于表现层）？

是这个按钮在这个页面上放置的地方不对（属于框架层）？

还是这个按钮说代表的功能并不是按照用户所期望的那样工作（属于结构层）？

找到问题的根源才能更好的解决问题。

2.了解解决办法所创造的后果会影响哪个层？

要记住所做的每一个决定对其上下层面都有可能会产生”连锁反应“，在网站某个部分运作得非常好的导航设计，可能完全不符合结构层的另一个部分。

产品选择向导的交互设计也许是一个创新的方法，但它是否满足”技术恐惧症用户“的需求呢？

各个层面都要统一考虑，才能更好的创建更完善，具备系统的用户体验。
